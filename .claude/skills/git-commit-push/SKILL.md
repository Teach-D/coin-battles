## 역할

staged된 변경사항을 읽고 적절한 커밋 타입과 설명을 선택해 커밋 메시지를 만든 뒤, commit → push 순서로 실행한다.

---

## 실행 순서

### 1. Staged 변경사항 파악

```bash
git diff --cached --stat
git diff --cached
```

staged 파일이 없으면 사용자에게 알리고 종료한다.

### 2. 커밋 타입 선택

변경 내용을 보고 아래 기준으로 타입을 고른다:

| 타입 | 사용 상황 |
|------|----------|
| `feat` | 새로운 기능 구현 |
| `fix` | 버그 수정 |
| `add` | 새 파일·리소스·설정 추가 (기능 변경 없음) |
| `refactor` | 동작 변경 없이 코드 구조 개선 |
| `docs` | 문서, 주석, README 변경 |
| `style` | 포맷, 들여쓰기 등 스타일만 변경 |
| `test` | 테스트 코드 추가·수정 |
| `chore` | 빌드 설정, 의존성, 패키지 업데이트 |

여러 타입이 섞여 있으면 **가장 핵심 변경**에 해당하는 타입 하나를 선택한다.

### 3. 커밋 메시지 작성

```
<타입>: <변경 내용 한 줄 요약>
```

이슈 번호가 있으면:
```
<타입>: <변경 내용 한 줄 요약> #<이슈번호>
```

**규칙**
- 설명은 명령형으로 짧게 — "~를 추가", "~를 수정", "~를 제거" 형태
- 변경한 파일 목록을 나열하지 않고, **왜/무엇을 했는지** 중심으로 작성
- 이슈번호는 사용자가 메시지나 인자로 제공한 경우에만 붙인다

### 4. 사용자에게 확인 요청

커밋 전에 메시지를 보여주고 확인을 받는다:

```
커밋 메시지: feat: 업비트 WebSocket 연동 추가 #42

진행할까요? (y/n)
```

사용자가 메시지 수정을 요청하면 반영한 뒤 다시 확인한다.

### 5. Commit → Push 실행

```bash
git commit -m "<커밋 메시지>"
git push
```

push 실패 시 (upstream 미설정 등) 적절한 명령어를 제안한다:
```bash
git push --set-upstream origin <현재 브랜치명>
```

---

## 이슈번호 인식 방법

사용자 메시지에서 아래 패턴으로 이슈번호를 추출한다:
- `#42`, `#이슈 42`, `이슈번호 42`, `issue 42`

명시적으로 없으면 붙이지 않는다. 추측하거나 물어보지 않는다.
