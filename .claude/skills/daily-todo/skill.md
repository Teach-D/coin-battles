# Daily TODO 스킬

사용자가 오늘 할 일을 요청하면, 기획서와 현재 프로젝트 상태를 분석해 현실적인 하루치 작업을 `TODO.md`에 추가합니다.

## 실행 순서

### 1단계: 현재 상태 파악 (병렬 실행)

아래 세 가지를 동시에 읽어 현황을 파악하세요:

1. **기획서 읽기** — `CoinBattle_MVP_기획서_v1.md`
2. **TODO.md 읽기** — 기존 완료 항목과 미완료 항목 확인
3. **코드 상태 확인** — Glob으로 `backend/src/**/*.kt`와 `frontend/src/**/*.tsx` 파일 목록 조회

### 2단계: 현재 Phase 확인

`CLAUDE.md`에서 현재 Phase를 확인하세요.

```
현재 Phase: Phase 1 (회원/인증, 업비트 시세, 매수/매도, 레버리지+숏, 기본 랭킹)
```

Phase별 우선순위:
- **Phase 1**: 회원/인증 → 업비트 WebSocket 시세 → 매수/매도 기본 → 레버리지/숏 → 기본 랭킹
- **Phase 2**: PVP 배틀 매칭 → 결과 카드 생성 → 소셜 공유
- **Phase 3**: 바이낸스 시세 통합 → 고급 랭킹 → 펀딩비/슬리피지

### 3단계: 오늘 할 일 결정

다음 기준으로 오늘 작업 3~6개를 선정하세요:

**선정 기준 (우선순위 순):**
1. TODO.md에 미완료로 남아있는 항목이 있으면 해당 작업 그룹을 먼저 배치
2. 코드 파일이 없거나 빈 상태인 기능을 Phase 우선순위대로 배치
3. 이미 구현된 코드가 있다면 다음 단계 작업을 배치

**작업 규모 및 영역 밸런스 기준:**
- 하루 작업량은 집중 4~6시간 기준으로 현실적으로 설정
- 작업 간 의존성을 고려 (예: Entity 없이 Repository 불가)
- **[영역 밸런스 및 누락 체크]**:
  - **기능 개발(Feature)** 성격의 작업인 경우: 백엔드 API와 프론트엔드 UI가 모두 필요한지 검토하세요.
  - 만약 백엔드 작업만 선정했다면, "이 기능이 완료되었을 때 유저가 실제로 사용할 수 있는 UI가 준비되었는가?"를 자문하고, 누락되었다면 프론트엔드 작업을 반드시 포함시키세요. (반대의 경우도 마찬가지)
  - **단순 수정/개선(Maintenance)** 성격의 작업(예: 버그 수정, 성능 최적화, 단순 UI 변경)인 경우에는 해당 영역의 작업만 배치합니다.

**CoinBattle 핵심 아키텍처 참고:**
- 백엔드 레이어: `domain` → `application` → `infrastructure` → `api`
- 동시성 방어: Redisson 분산 락 + 낙관적 락(@Version) + 멱등성 키
- 시세 흐름: 업비트 WebSocket → Redis Hash 캐싱 → Redis Pub/Sub → STOMP
- 주문 흐름: Redisson 락 → 잔고 검증 → DB 트랜잭션 → @Async 팬아웃

### 4단계: TODO.md 업데이트

`TODO.md`에 오늘 날짜 섹션을 추가하세요.

**형식:**

```markdown
## YYYY-MM-DD

### [영역] — [작업 그룹명]

- [ ] [구체적인 작업 항목]
- [ ] [구체적인 작업 항목]
- [ ] [구체적인 작업 항목]
```

**규칙:**
- 날짜는 `## YYYY-MM-DD` 형식 (오늘 날짜)
- 같은 날짜 섹션이 이미 있으면 해당 섹션에 새 작업 그룹을 추가 (기존 항목 건드리지 않음)
- 각 항목은 `- [ ]` 형식 (완료 시 `- [x]`)
- 항목명은 구체적으로: 파일명, API 경로, 메서드명 포함 권장
  - 좋음: `User 엔티티 설계 (User.kt, @Version 포함)`
  - 나쁨: `유저 관련 코드 작성`

**TODO.md 파일 위치**: 프로젝트 루트 `TODO.md`

기존 완료 항목(`- [x]`)은 절대 수정하지 않습니다.

### 5단계: 요약 출력

업데이트 후 오늘 추가된 작업 목록을 간결하게 출력하세요:

```
📋 오늘 할 일 (YYYY-MM-DD)

[영역]: N개 항목
- 항목1
- 항목2
...

TODO.md에 추가 완료
```

## 주의사항

- 이미 존재하는 `.kt` 또는 `.tsx` 파일이 있다면 해당 기능은 구현 완료로 간주하고 다음 작업으로 넘어가세요.
- 작업 항목이 너무 추상적이면 면접이나 PR 작성 시 쓸모없습니다. 항상 파일명/경로/메서드명을 포함해 구체적으로 작성하세요.
- 하루에 너무 많은 항목(7개 이상)을 넣지 마세요. 현실적인 완료 가능 수량으로 설정해 성취감을 줍니다.
- 사용자가 특정 기능을 언급했다면 그 기능을 오늘 작업의 중심에 배치하세요.
