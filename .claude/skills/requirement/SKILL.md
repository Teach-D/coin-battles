## 역할

사용자가 설명한 기능을 CoinBattle 아키텍처 맥락에서 분석하고,
도메인 모델 · 비즈니스 규칙 · 시나리오를 구조화된 요구사항 명세서로 작성한다.

> **api-spec 스킬과의 분업**
> - 이 스킬: 도메인 모델 / 비즈니스 규칙 / 상태 전이 / 시나리오 (WHAT & WHY)
> - api-spec 스킬: HTTP 엔드포인트 / DTO / 에러 코드 (HOW)
>
> 구현 전 흐름: `/requirement` → `/api-spec` → 백엔드/프론트엔드 구현

---

## 1단계: 기능 파악

명세를 쓰기 전에, 기능 설명에서 아래 항목을 추출한다.
명확하지 않은 것은 합리적인 기본값으로 채운다 — 사소한 것으로 먼저 묻지 않는다.

- **도메인**: auth / market / order / position / battle / ranking / card / notification 중 어디?
- **핵심 행위자**: USER (인증된 플레이어), ADMIN, 시스템(스케줄러/이벤트)
- **상태 전이 여부**: 명확한 status 필드가 있는가?
- **동시성 위험**: 여러 사용자가 동시에 같은 자원을 변경하는가?
- **실시간 요구**: WebSocket 브로드캐스트가 필요한가?
- **외부 의존**: 업비트/바이낸스 시세, Redis, 스케줄러 등 외부 시스템 연동 여부

## 2단계: 명세 작성

아래 7개 섹션을 순서대로 작성한다.
해당 없는 섹션은 "해당 없음"으로 표기하지 않고 **섹션 자체를 생략**한다.

---

### 섹션 1 — 개요

```markdown
## 1. 개요

- **기능 목적**: {한 문장으로 이 기능이 CoinBattle에서 무엇을 해결하는지}
- **핵심 사용자**: {USER / ADMIN / 시스템}
- **범위**
  - In Scope: {포함}
  - Out of Scope: {제외 — 특히 "실제 결제", "외부 거래소 실주문" 등 명확히}
```

---

### 섹션 2 — 도메인 모델

```markdown
## 2. 도메인 모델 후보

### 엔티티 목록

| 엔티티 | 핵심 속성 | 기존 엔티티 참조 |
|--------|-----------|-----------------|
| `{EntityName}` | id, ..., status, createdAt | {기존 Battle / Order / Position 등과 관계} |

### 엔티티 간 관계

- `{A}` 1 ↔ N `{B}` — {관계 설명}
- `{X}` N ↔ 1 `User` — {관계 설명}

### 상태 다이어그램 — {Entity}.status   (상태 필드가 있는 경우만)

{STATE_A} → {STATE_B} ({전이 조건})
{STATE_A} → {STATE_C} ({전이 조건})
{STATE_B} → (종료, 변경 불가)
```

CoinBattle 기존 엔티티와의 정합성을 확인한다:
- `Battle`: WAITING → IN_PROGRESS → FINISHED
- `Order`: PENDING → FILLED / CANCELLED
- `Position`: OPEN → CLOSED (청산 or 수동 종료)

---

### 섹션 3 — 비즈니스 규칙

```markdown
## 3. 비즈니스 규칙

1. **BR-01** {규칙 제목}
   - 조건: {언제 적용}
   - 위반 시: {HTTP 상태 코드} {ErrorCode}

2. **BR-02** ...
```

CoinBattle 공통 비즈니스 규칙 (기능과 관련 있으면 참조):

| 규칙 | 내용 |
|------|------|
| 레버리지 범위 | 1x ~ 10x, 위반 시 400 INVALID_LEVERAGE |
| 강제청산 기준 | 손실률 = -(1 / 레버리지) × 0.9 |
| 슬리피지 | 100만 이하: 없음 / 100~500만: ±0.05% / 500만~전액: ±0.1~0.3% |
| 배틀 체결 기준 | 서버 수신 시각 기준 (클라이언트 시각 무시) |
| 주문 잔고 검증 | 주문 금액 ≤ 보유 잔고 × 레버리지 |
| 동시 주문 | Redisson 분산 락 `user:{id}:order` TTL 3초로 직렬화 |

## 3단계: 작성 스타일

- 한국어로 설명, 엔티티명/필드명/에러코드는 영어
- 비즈니스 규칙은 **BR-NN** 형식, 위반 시 ErrorCode 반드시 명시
- 아직 결정되지 않은 사항은 `TBD` 또는 `미결 사항` 섹션으로 분리
- 기존 CoinBattle 엔티티(Battle, Order, Position, User)와의 연관을 명확히 표기
- 동시성이 필요한 경우 CLAUDE.md의 3단계 방어 패턴(분산 락 → 낙관적 락 → 멱등성 키)을 참조해 적용 수준 명시
